AI Fundamentals & Cursor Basics

Understanding how large language models actually work, why that matters for how you use them, and building a practical toolkit of prompt engineering techniques for Java and React development.

Understanding Large Language Models in a Development Context

What LLMs Actually Are

At their core, large language models are neural networks built on the transformer architecture. They work by predicting the most likely next token in a sequence. That is the entire mechanism -- there is no hidden compiler, no secret reasoning engine, no database of code snippets being looked up. It is next-token prediction, trained on enormous amounts of text.

The model takes in a sequence of tokens, passes them through layers of attention mechanisms (which figure out which parts of the input are most relevant to each other), and outputs a probability distribution over what comes next. It samples from that distribution, appends the result, and repeats.

Talk track: "When you ask Cursor to write a Spring Boot controller, it's not compiling Java in its head. It's not looking at a database of controllers and finding a match. It has seen millions of Spring Boot controllers during training, and it's generating one token at a time -- @RestController, then public, then class -- each time predicting 'what's the most likely next piece of code given everything that came before?' The result often looks like it understands Java. But understanding and statistical prediction are different things, and that distinction matters for how you use these tools."

How Training Data Shapes Capabilities

The model's abilities are a direct reflection of its training data. GitHub Copilot was trained heavily on public GitHub repositories, which means it excels at patterns that appear frequently in open-source code. Claude (which Cursor also supports) was trained on a broader mix of text -- books, websites, documentation, code -- which gives it stronger reasoning about intent and architecture, but sometimes less precise syntax recall for niche frameworks.

  • Heavily-represented patterns (Spring Boot REST controllers, React hooks, Express routes) get excellent generation quality
  • Under-represented patterns (proprietary internal frameworks, obscure libraries, very new APIs) produce more errors and hallucinations
  • Your specific codebase conventions are not in the training data -- you need to provide that context explicitly

The "Stochastic Parrot" vs. "Reasoning Engine" Debate

This is a real debate in AI research, and it has practical implications for your daily work. The "stochastic parrot" view says LLMs are sophisticated pattern matchers that recombine training data without genuine understanding. The "emergent reasoning" view says that at sufficient scale, something resembling actual reasoning emerges from the patterns.

For your purposes as developers, the answer is: it doesn't matter which is true. What matters is how you work with the tool.

Key Insight

Treat the AI like a very well-read junior developer who has seen an enormous amount of code but has never actually shipped a production system. They can write code that looks correct, follows common patterns, and passes a first-glance review. But they have no real understanding of your specific business domain, your deployment environment, or the subtle requirements that live in your team's heads but not in the codebase.

Token Limits, Context Windows, and Impact on Code Generation

What Tokens Are

Tokens are subword units -- the fundamental pieces the model works with. They are not characters, not words, but something in between. In English prose, a token averages about 4 characters. But code is more token-dense because of camelCase identifiers, special characters, and structured syntax.

English text

// "The quick brown fox" = 4 tokens // ~4 characters per token // 1 page of English ~ 250 tokens

Java code

// "productRepository.findById(id)" = ~8 tokens // Code averages ~2-3 chars per token // A 100-line Java class ~ 500-800 tokens

A 200k context window sounds enormous, and it is. But a medium-sized Java/React project can easily be 50,000+ lines of code. You are never going to fit your entire codebase in context. And even if you could, you would not want to -- quality degrades with very long contexts.

Talk track: "Think of the context window as the model's working memory. Just like you can't hold an entire codebase in your head at once, the model can't either. And just like your own memory, the things at the beginning and end of what it's 'reading' tend to get more attention than stuff in the middle. This is actually a well-documented phenomenon called the 'lost in the middle' problem -- research shows models pay significantly less attention to information placed in the middle of long contexts compared to the beginning and end."

The "Lost in the Middle" Problem

Research from Stanford and others has shown that when you give an LLM a very long context, it pays the most attention to:

  1. The very beginning of the context (system prompt, first instructions)
  2. The very end of the context (your most recent message, the most recent code)
  3. Content in the middle gets significantly less attention, leading to missed details and instructions

How Cursor Manages Context

This is one of the key reasons Cursor exists as a product rather than just using ChatGPT with copy-paste. Cursor provides several mechanisms to keep context focused and relevant:

  • Workspace indexing -- Cursor indexes your entire codebase locally, creating embeddings that allow it to find relevant code quickly
  • RAG (Retrieval-Augmented Generation) -- When you ask a question, Cursor retrieves the most relevant code snippets and includes them in context, rather than dumping your entire codebase
  • @ references -- You can explicitly point the model at specific files (@ProductService.java), folders (@src/main/java), or even the whole codebase (@codebase)
  • Automatic context selection -- In Agent mode, Cursor determines which files to read based on the task

Key Insight

The best way to get high-quality output from any model is not to maximize context -- it's to optimize context. Give the model exactly what it needs and nothing more. A focused 2,000-token context will almost always produce better results than a sprawling 100,000-token context that includes irrelevant files.

Prompt Engineering Essentials for Technical Tasks

Zero-Shot Prompting

The simplest technique: just tell the model what you want. No examples, no elaborate setup. This works well when the task is straightforward and the model has seen many similar examples in training.

// Zero-shot: Direct instruction "Write a Java method that validates an email address using regex. Return true if valid, false otherwise."

Zero-shot works best for well-known patterns. The model has seen thousands of email validators. But for anything specific to your project, you will need more context.

Few-Shot Prompting

Provide input/output examples to show the model the pattern you want. This is especially powerful when your project follows specific conventions that differ from common open-source patterns.

// Few-shot: Provide examples to guide behavior "Convert these REST endpoints to follow our project pattern: Example input: GET /users/{id} Example output: @GetMapping("/users/{id}") public ResponseEntity<ApiResponse<UserDTO>> getUser( @PathVariable Long id) { return ResponseEntity.ok(ApiResponse.success(userService.findById(id))); } Now convert: POST /products with body ProductCreateRequest"

The model sees the pattern (custom ApiResponse wrapper, ResponseEntity, DTO naming convention) and applies it to the new endpoint. Without the example, it would generate a generic Spring Boot controller.

Chain-of-Thought Prompting

Ask the model to reason step by step. This dramatically improves results on complex tasks like debugging, architecture decisions, and multi-step refactoring.

// Chain-of-thought: Step-by-step reasoning "This React component re-renders excessively. Think step by step: 1. First, identify all state variables and where they change 2. Then, trace which props are passed to child components 3. Check for missing memoization or unstable references 4. Suggest specific fixes with code examples Here is the component: @ProductList.tsx"
Talk track: "Chain-of-thought is your most powerful debugging technique. When you just say 'fix this bug,' the model often jumps to the most common cause and may miss the actual issue. When you say 'think step by step about why this could be happening,' it systematically works through possibilities. I've seen this turn a 50% success rate on debugging prompts into closer to 85%."

Role Prompting

Set a persona that shapes the model's response style and priorities. This is more impactful than many people expect.

For backend work

"You are a senior Java developer with 10 years of Spring Boot experience. You prioritize: - Constructor injection over field - Proper exception handling - Meaningful logging - Thread safety"

For frontend work

"You are a senior React developer who follows modern best practices: - Functional components only - Custom hooks for shared logic - TypeScript strict mode - Accessibility (WCAG 2.1 AA)"

The Anatomy of a Great Prompt

The best prompts combine four elements. Each one is optional, but the more you include, the better your results:

  1. Context

    What is the project? What files are relevant? What has been tried before? Use @file references and describe the architecture.

  2. Task

    What specifically do you want? Be precise. "Add error handling" is vague. "Add try-catch blocks to all service methods that call the repository layer, returning appropriate HTTP status codes" is specific.

  3. Constraints

    What rules must be followed? "Use constructor injection, not field injection." "Keep methods under 20 lines." "Don't add new dependencies." "Follow the existing patterns in @UserService.java."

  4. Output Format

    What should the response look like? "Show the complete file." "Just show the changed methods." "Include unit tests." "Explain your changes in comments."

Writing Effective Prompts for Java and React Development

Java-Specific Prompt Patterns

Java and Spring Boot have strong conventions. Referencing them explicitly produces dramatically better results.

Reference the Framework

"Follow Spring Boot 3.x conventions. Use @Service, @Repository, @RestController. Use constructor injection (not @Autowired on fields)."

Reference Existing Code

"Follow the same patterns used in @ProductService.java and @ProductController.java for the new CartService."

Specify Exception Handling

"Use @GlobalExceptionHandler pattern from @GlobalExceptionHandler.java. Throw ResourceNotFoundException for missing entities."

React-Specific Prompt Patterns

React has evolved significantly. Being explicit about which patterns you want prevents the model from generating class components or outdated practices.

Specify Component Style

"Use functional components with hooks. Use TypeScript with strict types. Export named components, not default."

Reference Project Patterns

"Follow the existing patterns in @ProductList.tsx for data fetching. Use our CartContext from @CartContext.tsx for state management."

Style Conventions

"Use CSS modules with BEM naming. Match the styling approach in @ProductCard.module.css. Include responsive breakpoints."

Talk track: "The difference between 'make a controller' and a well-crafted prompt is the difference between throwaway code and production-quality output. The vague prompt might give you something that compiles, but you'll spend 30 minutes fixing it. The structured prompt gives you something that's 90% ready for a PR. Over a full day of development, that difference compounds enormously."

Common Mistakes to Avoid

  • Being too vague -- "Fix the bug" tells the model nothing. "The addToCart method throws a NullPointerException when the product ID doesn't exist in the database" gives it what it needs.
  • Not referencing existing code -- If you don't use @file references, the model generates generic code that doesn't match your project's patterns.
  • Omitting constraints -- Without constraints, the model makes assumptions. It might use field injection when you want constructor injection, or add a new dependency when you want to use existing ones.
  • Asking for too much at once -- "Build the entire checkout flow" is too broad. Break it into steps: first the service, then the controller, then the tests.
  • Ignoring the output format -- If you don't specify, the model might explain the code instead of writing it, or write incomplete snippets instead of full files.

Where AI Adds Value vs. Where Human Judgment is Critical

Where AI Excels

Boilerplate Code

CRUD controllers, DTOs, entity classes, configuration files. The model has seen thousands of these and generates them accurately.

Common Patterns

Authentication middleware, pagination, sorting, filtering, error handling. Well-established patterns it has seen many times.

Refactoring

Extracting methods, renaming variables, converting between patterns (callbacks to async/await, class to functional components). Mechanical transformations.

Test Generation

Unit tests for existing code, especially happy-path cases. It can see the method signature and generate sensible test cases quickly.

Documentation

Javadoc, JSDoc, README files, API documentation. It can read code and explain it clearly -- often better than the developer who wrote it.

Translation Between Languages

Converting a Python script to Java, or a JavaScript snippet to TypeScript. Cross-language patterns it has seen in training data.

Where AI Struggles

Novel Business Logic

Your specific domain rules, edge cases that come from real-world usage, business requirements that aren't captured in code. The model has never seen YOUR business.

Security Decisions

Authentication architecture, authorization rules, data privacy policies, compliance requirements. Getting security wrong has real consequences.

Architecture Choices

Monolith vs. microservices, database selection, caching strategy, API design philosophy. These depend on your specific scale, team, and constraints.

Your Specific Domain

Internal APIs, proprietary libraries, organizational conventions, deployment-specific requirements. None of this was in the training data.

The 80/20 Rule of AI-Assisted Development

AI gets you 80% of the way there. It generates the structure, fills in the boilerplate, follows the patterns. The remaining 20% -- review, edge cases, integration, domain-specific logic -- is where developers earn their keep.

AI Generates (80%) Developer Completes (20%) ========================== ========================== Basic structure/scaffolding Edge case handling Common patterns Business logic validation Boilerplate code Security review Standard error handling Performance optimization Initial test cases Integration testing Documentation drafts Architecture decisions
Talk track: "The 80/20 breakdown is real, and it's why AI makes experienced developers more productive rather than replacing them. A junior developer using AI might accept the 80% as-is and ship buggy code. A senior developer uses AI for the 80% and then applies their judgment to the remaining 20% -- that's where years of experience matter. The review, the edge cases, the 'wait, what about concurrency?' moments. That's the 20% that separates production code from demo code."

Critical Warning

Never ship AI-generated code without understanding it. If you can't explain what every line does, you shouldn't commit it. AI can introduce subtle bugs -- incorrect null handling, missing thread safety, security vulnerabilities that look correct at first glance. Treat every AI output like a pull request from a junior developer: read it carefully, test the edge cases, and verify it actually solves the right problem.

The Reviewer Mindset

The most productive way to use AI coding assistants is to adopt the reviewer mindset. Instead of writing code and then reviewing it, you are reviewing code as it's generated.

  • Read every line -- Don't just check if it compiles. Understand the logic.
  • Question assumptions -- Why did it use this approach? Is there a simpler way?
  • Check edge cases -- What happens with null input? Empty lists? Concurrent access?
  • Verify error handling -- Does it fail gracefully? Are errors logged? Are user-facing messages appropriate?
  • Test the boundaries -- The AI's test cases often only cover happy paths. Add the adversarial ones yourself.

Model Selection Strategies

When to Use Which Model

Cursor gives you access to multiple models, and choosing the right one for each task can save time and improve quality. The key tradeoff is always speed vs. quality vs. cost.

Scenario Recommended Model Why
Quick autocomplete, simple edits Fast model (GPT-4o mini, Claude Haiku) Speed matters more than depth. Simple tasks don't need large models.
Code generation, standard features Mid-tier (GPT-4o, Claude Sonnet) Good balance of quality and speed for everyday coding tasks.
Complex debugging, architecture Top-tier (Claude Opus, o1/o3) Needs deep reasoning, understanding of complex interactions.
Multi-file refactoring Claude Sonnet / GPT-4o Needs large context window and good code understanding.
Writing tests from specifications Claude Sonnet Good at understanding intent and generating comprehensive test cases.
Explaining legacy code Claude Opus / Sonnet Strong reasoning about code intent and documentation generation.

Cursor's Model Selector

In Cursor, you can switch models at any time during a chat session. The model selector is in the top-right of the Chat panel. The practical advice:

  • Start with a fast model for exploration and simple tasks
  • Switch to a powerful model when you hit a complex problem or need higher quality output
  • Use the fast model for Tab completions -- you want instant suggestions, not deep analysis
  • Use a powerful model for Agent mode -- multi-step tasks need stronger reasoning to stay on track

Key Insight

Don't use a top-tier model for everything. It's slower and uses more of your Cursor quota. Reserve the powerful models for tasks that genuinely need them: complex debugging, architecture decisions, multi-file changes with tricky dependencies. For boilerplate, standard patterns, and simple edits, the fast models are equally capable and much quicker.

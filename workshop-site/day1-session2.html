Introduction to Cursor IDE

From workspace indexing to Agent Mode — a comprehensive tour of every interaction surface in Cursor. This session builds the muscle memory you will use for the rest of the workshop.

1. Existing Codebase Setup & Navigation

"When you first open a project in Cursor, it starts building a map of your codebase. You will see a progress indicator in the status bar — that is the indexing engine scanning every file, generating embeddings, and storing them locally so the AI can search your code semantically rather than just by keyword. This is the foundation that makes everything else — Chat, Composer, Agent Mode — actually useful."

How Workspace Indexing Works

  • First-time indexing — Cursor walks the file tree, reads every source file, and produces vector embeddings (dense numeric representations of meaning). This can take a few minutes on a large repo.
  • Incremental updates — After the initial pass, only changed files are re-indexed. Saves happen automatically on file save.
  • What gets indexed — Source code, config files, markdown docs. Binary files and anything in .gitignore are skipped by default.
  • Where it lives — The index is stored in .cursor/ locally. It is not sent to the cloud.

Context Generation: Embeddings & Semantic Search

"Context generation" is the process of turning your code into something the language model can reason about. Cursor converts each file (and often smaller chunks within files) into embeddings — high-dimensional vectors that capture semantic meaning. When you ask a question, Cursor converts your question into an embedding too, and then finds the most similar code chunks using vector similarity search.

This is fundamentally different from grep. A semantic search for "cart pricing logic" will find methods like calculateSubtotal() even though the word "pricing" never appears in the code.

How @codebase Works Behind the Scenes (RAG)

When you type @codebase in Chat, Cursor uses Retrieval Augmented Generation (RAG):

  1. Your question is embedded into a vector.
  2. The vector index is queried for the top-N most relevant code chunks.
  3. Those chunks are injected into the prompt as context, alongside your question.
  4. The LLM generates an answer grounded in your actual code.
Your Question | v [ Embed Query ] ---> [ Vector Index ] ---> Top-N Code Chunks | v [ LLM Prompt = Question + Chunks ] | v Grounded Answer

Live Demo

Let's open our ShopCursor project and watch the indexing status. Open Cursor, then File → Open Folder and select the ShopCursor root. Watch the status bar at the bottom — you will see "Indexing..." with a progress percentage. Once it finishes, try Cmd+L and ask: "What does this project do?" using @codebase.

2. Chat Mode for Codebase Exploration

"Chat is your conversation partner for the codebase. Think of it as pair programming with someone who has read every file in the project. You open it with Cmd+L, ask a question, and it answers with references back to the actual source. The real power comes from the @ reference system — you can point the AI at specific files, folders, symbols, or the entire codebase."

Opening Chat & Asking Questions

  • Cmd+L opens the Chat panel on the right side.
  • Type a natural language question — "How does authentication work in this app?"
  • Chat will search the codebase for relevant context and give you an answer with code references.
  • Click any file reference in the response to jump directly to that code.

The @ Reference System

@ references are how you give the AI precise context. Instead of hoping it finds the right files, you tell it exactly what to look at.

Reference What It Does Example
@file Includes a specific file in context @CartService.java explain this service
@folder Includes all files in a directory @backend/src/main/java/ list all controllers
@symbol References a specific class, function, or variable @addToCart what does this method do?
@codebase Searches entire indexed codebase via RAG @codebase how is error handling structured?
@web Searches the web for current information @web latest Spring Boot 3.2 migration guide
@docs Searches indexed documentation (configured per project) @docs how to configure CORS in Spring?
@git References git history, diffs, and commits @git what changed in the last 3 commits?
@terminal Includes recent terminal output @terminal why did that test fail?

Multi-Turn Conversations

Chat remembers the full conversation history within a session. You can ask a follow-up question without repeating context — "Now show me the tests for that service" works because Chat knows which service you were just discussing.

Key Insight

When to use Chat vs. other modes: Chat is best for exploration and learning — understanding how code works, asking "why" questions, getting explanations of architecture. When you are ready to make changes, switch to Composer or Agent Mode.

Live Demo

Open Chat (Cmd+L) and type:

@CartService.java Explain how this service manages cart state. What data structure holds the cart items, and how are quantities updated?

Then follow up: "What happens if I try to add a product that doesn't exist?"

3. Composer for Multi-File Edits

"Composer is where Cursor really starts to feel like a team member rather than an autocomplete. When you need to add a feature that touches the database layer, the service, the controller, and the frontend — that is a Composer job. It looks at all the files involved, plans the changes, and then applies them in one coordinated batch. You review the diffs, accept or reject, and you are done."

What Makes Composer Different

  • Multi-file orchestration — Composer can create, edit, and delete multiple files in a single pass.
  • Planning before executing — It first outlines what it intends to do, then generates the code. You see the plan before any files change.
  • Diff-based review — Every change is shown as a diff you can accept or reject per-file.
  • Context-aware — It reads your existing code patterns, naming conventions, and project structure to match them.

Opening & Using Composer

  1. Press Cmd+I to open the Composer panel.
  2. Describe what you want in natural language — be specific about which layers to touch.
  3. Use @ references to point Composer at relevant existing files as examples.
  4. Review the proposed plan and generated diffs.
  5. Accept all, accept per-file, or reject and refine your prompt.

When to Use Composer

Good Fit

  • Full-stack feature spanning backend + frontend
  • Adding a new API endpoint with service, controller, and tests
  • Cross-cutting concerns (logging, error handling across files)
  • Scaffolding a new module with multiple files

Not the Best Fit

  • Single-method refactoring (use Cmd+K instead)
  • Exploratory questions (use Chat instead)
  • Complex multi-step tasks with iteration (use Agent Mode)
  • Changes you are not sure about yet (explore in Chat first)

Live Demo

Open Composer (Cmd+I) and enter:

"Add a Product Reviews feature. Create a Review entity with rating (1-5) and comment fields, a ReviewRepository, a ReviewService, and a ReviewController with GET and POST endpoints at /api/products/{id}/reviews. Follow the same patterns as the existing Product entity and controller. @Product.java @ProductController.java"

Watch how Composer plans the file structure, then generates each file matching your existing conventions.

4. Inline Editing for Targeted Refactoring

"Inline editing is the surgical tool. You select a chunk of code, press Cmd+K, describe what you want in plain English, and Cursor rewrites just that selection. It shows you a diff right there in the editor — green for additions, red for removals — and you hit Tab to accept or Escape to reject. It is the fastest way to make targeted changes."

The Cmd+K Workflow

  1. Select the Code

    Highlight the method, block, or lines you want to change. The more precise your selection, the better the result.

  2. Press Cmd+K

    An inline prompt bar appears. Describe the change in natural language.

  3. Review the Diff

    Cursor shows a side-by-side diff of the proposed changes directly in the editor.

  4. Accept or Reject

    Press Tab to accept, Esc to reject. You can also partially accept with Cmd+Y.

Best Use Cases

  • Single-method refactoring — "Extract the validation logic into a private method"
  • Adding error handling — "Wrap this in a try-catch and throw a custom exception"
  • Quick fixes — "Add null checks for all parameters"
  • Type conversion — "Convert this to use Optional instead of null"
  • Adding validation — "Add input validation for quantity and productId"

Tips for Getting Good Results

Be Specific

Instead of "make this better", say "add input validation for quantity (must be positive) and productId (must not be null), throwing IllegalArgumentException with descriptive messages."

Select Precisely

Select only the code you want changed. If you select the entire file, Cmd+K might rewrite things you did not intend to change.

Iterate

If the first result is not right, reject it (Esc) and refine your prompt. Add constraints like "keep the method signature the same" or "do not change the return type".

Live Demo

Open CartService.java, select the addToCart method body. Press Cmd+K and type:

"Add input validation: quantity must be > 0 and <= 99, productId must not be null. Throw IllegalArgumentException with clear messages."

Review the diff, accept with Tab.

5. Agent Mode: Autonomous Multi-Step Execution

"Agent Mode is the big upgrade in Cursor 2.0. Instead of you driving every step, the AI takes your goal and works toward it autonomously — reading files, writing code, running commands, fixing errors, and iterating until the task is done. It is the default mode now, and it is where Cursor becomes genuinely transformative for your workflow."

The Plan → Execute → Iterate Cycle

When you give Agent Mode a task, it follows a structured approach:

  1. Plan — The agent reads relevant files, understands the codebase, and outlines a plan of action.
  2. Execute — It creates and edits files, runs shell commands (with your approval), and writes tests.
  3. Iterate — If something fails (a test, a build, a linting error), the agent reads the error, diagnoses the issue, and tries again.
[ Your Goal ] | v [ PLAN ] <--- reads files, analyzes codebase | v [ EXECUTE ] <--- edits files, runs commands | v [ CHECK ] <--- tests pass? build succeeds? | / \ v v [Done] [Fix & Retry] ---> back to EXECUTE

Checkpoints: Safe Rollback

Before every edit, Agent Mode creates an automatic checkpoint — a snapshot of the file state. If the agent goes down a wrong path, you can roll back to any previous checkpoint with one click. This makes Agent Mode safe to use even for experimental tasks.

Key Insight

Checkpoints are not git commits. They are lightweight, in-memory snapshots that live within your Cursor session. If you like the result, commit it to git. If you do not, roll back and the files are restored instantly.

Background Agents

  • Run up to 8 parallel background agents, each working on a separate task.
  • Each background agent operates in a git worktree — an isolated copy of your repo. No conflicts with your working directory.
  • When a background agent finishes, it creates a branch with its changes. You review and merge like any PR.
  • Great for: "Add tests for all services" + "Update all deprecated API calls" + "Add logging to controllers" — all running simultaneously.

Subagents

Agent Mode can delegate subtasks to specialized subagents:

Explore

Reads files and searches the codebase to gather information. Read-only — it never modifies files. Perfect for understanding code before making changes.

Bash

Executes shell commands — running tests, installing packages, building the project. The agent uses output to verify its work.

Browser

Opens a headless browser to interact with web pages. Useful for testing frontend changes, scraping documentation, or verifying API endpoints.

You can also create custom subagents in .cursor/agents/. These are markdown files with YAML frontmatter that define a specialized agent with its own model, permissions, and instructions.

Live Demo

In Agent Mode (the default), type:

"Add a new GET endpoint at /api/products/featured that returns the top 5 products ordered by name. Create the repository method, service method, and controller endpoint. Write a unit test for the service method. Follow existing patterns in @ProductService.java and @ProductController.java."

Watch the agent plan its approach, create each layer, and then run the tests to verify.

6. Automated Code Review & Refactoring

"One of the most practical uses of Cursor is code review. You can paste code into Chat, reference a file with @, or use a slash command — and ask the AI to review it for specific issues. It is not a replacement for human review, but it catches things that humans routinely miss: subtle race conditions, missing null checks, SQL injection vectors."

Review Approaches

  • Chat review — Open Chat, reference a file with @filename, and ask: "Review this file for security issues and anti-patterns."
  • The /review pattern — Create a custom command at .cursor/commands/review.md that defines your team's review checklist. Then just type /review in agent chat.
  • Agent-driven review — In Agent Mode, ask: "Review all files changed in the last commit and create a summary of issues."

Targeted Review Types

Security Review

  • SQL injection and XSS vulnerabilities
  • Authentication / authorization gaps
  • Hardcoded secrets or credentials
  • Input validation weaknesses

Quality Review

  • God classes and long methods
  • Code duplication
  • Missing error handling
  • Naming convention violations

BugBot for Automated PR Review

BugBot is Cursor's automated PR reviewer. When enabled on a repository, it automatically reviews every pull request and leaves comments on potential issues. Each comment includes a "Fix in Cursor" button that opens Cursor with the issue pre-loaded so you can fix it immediately.

7. Debugging & Error Analysis

"Here is a workflow that will save you hours: when you get an error, instead of reading the stack trace line by line, paste it into Chat. Cursor will map the stack trace to your actual code, identify the root cause, and suggest a fix. You can also use @terminal to pull in the most recent terminal output automatically."

Debugging Workflows

  1. Paste the Error

    Copy the error message or stack trace and paste it directly into Chat. Cursor correlates the line numbers and file names with your project.

  2. Use @terminal

    If the error is in your terminal, just type @terminal in Chat — it automatically includes the recent terminal output.

  3. Chain-of-Thought Debugging

    Ask: "Walk through this error step by step. What is the execution path that leads to this exception?" This forces the AI to trace the logic methodically.

  4. Apply the Fix

    Once Chat identifies the issue, ask it to generate the fix. You can apply it directly from Chat or copy it into Cmd+K for more control.

Live Demo

Paste this sample stack trace into Chat:

java.lang.NullPointerException: Cannot invoke "String.length()" at com.shopcursor.service.CartService.addToCart(CartService.java:47) at com.shopcursor.controller.CartController.addItem(CartController.java:32) at sun.reflect.NativeMethodAccessorImpl.invoke0(Native Method)

Ask: "Diagnose this NullPointerException. What is null at CartService.java:47, and what is the fix?"

8. Test Generation & Documentation

"AI-generated tests are one of the highest-value use cases. You can point Cursor at a service class and ask it to generate comprehensive JUnit tests, and it will cover happy paths, edge cases, and error scenarios. But — and this is important — you must review them. AI tests can look correct while testing the wrong thing."

Generating Tests

  • JUnit tests — "Generate JUnit 5 tests for @CartService.java. Cover happy path, empty cart, invalid product ID, and quantity edge cases."
  • Playwright E2E tests — "Generate Playwright tests for the cart page. Test adding items, updating quantity, and removing items. Follow the Page Object Model pattern in @e2e/pages/."
  • Following existing patterns — Use @ references to point at existing test files: "Follow the test style in @ProductServiceTest.java".

Documentation Generation

  • JavaDoc — Select a class or method, Cmd+K: "Add comprehensive JavaDoc including @param, @return, and @throws tags".
  • JSDoc — Same workflow for TypeScript/JavaScript: "Add JSDoc with type annotations and examples".
  • README generation — "Generate a README.md for this project based on the codebase structure and existing docs."

Warning

Always review generated tests. Common pitfalls:

  • Tests that assert on the implementation rather than behavior (brittle tests).
  • Mock setups that do not reflect real-world conditions.
  • Missing edge cases that the AI did not think of.
  • Tests that pass but do not actually test what you think they test (vacuous assertions).

9. Workspace Indexing & .cursorrules Configuration

"Rules are how you teach Cursor about your project's standards. Instead of repeating 'use constructor injection, not field injection' in every prompt, you write it once as a rule and it applies automatically to every AI interaction. Think of rules as your project's style guide for the AI."

The Legacy: .cursorrules

The original .cursorrules file (placed at the project root) is still supported but deprecated. It was a single file with instructions that applied globally. The problem: it scaled poorly for large projects with different conventions in different areas.

The Modern Replacement: Project Rules

Project Rules live in .cursor/rules/*.mdc. Each rule file has YAML frontmatter that controls when and how it is applied.

--- description: "Java service layer conventions" globs: ["backend/**/*Service.java"] alwaysApply: false --- # Service Layer Rules - Use constructor injection, never field injection - Services must be annotated with @Service - All public methods must have JavaDoc - Throw custom exceptions, never generic ones - Use Optional for nullable return types

The Four Rule Application Modes

Mode How It Works Best For
Always Apply alwaysApply: true — included in every prompt Universal standards (naming, formatting)
Glob-based globs: ["*.java"] — auto-included when matching files are in context Language or layer-specific rules
Intelligent Agent reads the description field and decides if the rule is relevant Domain-specific patterns
Manual User explicitly @-mentions the rule to include it Rarely used specialized guidelines

Key Insight

Rules shape every AI interaction in your project. A well-crafted set of rules means your entire team gets consistent AI output without each developer having to write detailed prompts. Commit your .cursor/rules/ directory to version control so the whole team benefits.

10. Essential Keyboard Shortcuts & Productivity Tips

"You do not need to memorize all of these right now. The three that will change your workflow today are Tab to accept, Cmd+K for inline edits, and Cmd+L for Chat. Everything else you will pick up as you use the tool."

Core Shortcuts

Shortcut Action When to Use
Tab Accept AI suggestion / autocomplete While typing — accept the ghost text
Esc Reject AI suggestion When the suggestion is wrong
Cmd+K Inline editing (select code first) Targeted single-block refactoring
Cmd+L Open Chat panel Ask questions, explore code
Cmd+I Open Composer Multi-file edits and feature scaffolding
Alt+] Next suggestion Cycle through alternative completions
Alt+[ Previous suggestion Go back to a previous completion option
Cmd+Shift+L Toggle AI panel Show/hide the AI sidebar

Productivity Tips

Start Fresh After Features

After completing a feature, start a new Chat conversation. Old context can confuse the AI when you switch tasks.

Use Targeted @ Refs

Prefer @file over @codebase when you know which files are relevant. It is faster and more accurate.

Keep Context Focused

Do not dump your entire project into context. The LLM has a limited window — give it what it needs, not everything.

Iterate, Do Not Restart

If the AI gets close but not perfect, refine your prompt rather than starting over. Say "Almost, but change X to Y."

11. Creating Skills & Commands

"Commands and Skills are how you encode your team's workflows so they are repeatable. A command is a one-liner — type /review and the AI knows exactly what to check. A skill is a multi-step recipe — type /add-endpoint and it walks through creating the entity, repository, service, controller, and tests in order."

Slash Commands

Commands are plain markdown files in .cursor/commands/*.md. You invoke them by typing / followed by the filename (without the extension).

# .cursor/commands/review.md Review the selected code or referenced file for: 1. Security - SQL injection, XSS, auth gaps 2. Performance - N+1 queries, unnecessary allocations 3. Patterns - Does it follow our project conventions? 4. Error handling - Are all failure modes covered? Provide a numbered list of findings with severity (critical / warning / suggestion) and code fixes.

Skills: Multi-Step Workflows

Skills live in .cursor/skills/<name>/SKILL.md. They have YAML frontmatter and can include supporting scripts, templates, and documentation in their directory.

# .cursor/skills/add-endpoint/SKILL.md --- name: add-endpoint description: "Scaffold a new REST endpoint with all layers" --- ## Steps 1. Ask the user for the entity name, fields, and HTTP methods 2. Create the JPA entity in backend/src/main/java/.../model/ 3. Create the repository interface 4. Create the service class with business logic 5. Create the controller with proper annotations 6. Generate JUnit tests for the service layer 7. Run ./mvnw test to verify everything compiles

When to Use Commands vs. Skills

Commands

  • Single prompt, single action
  • Plain markdown, no frontmatter needed
  • Best for: review, explain, format, lint
  • Quick to create, quick to invoke

Skills

  • Multi-step workflows with ordering
  • YAML frontmatter with name and description
  • Can include scripts and template files
  • Best for: scaffold feature, migrate pattern, setup config

Creating Your Own

  • /create-skill — Interactive wizard that scaffolds a new skill directory with SKILL.md.
  • /create-rule — Interactive wizard that creates a new .mdc rule file with frontmatter.
  • Or just create the files manually — the format is simple markdown with optional YAML frontmatter.

Live Demo

In agent chat, type /review and then reference @CartService.java. Watch how the command's instructions guide the AI through a structured review.

Then try: /create-skill to see the interactive skill creation wizard in action.

MCP Architecture, Skills & Commands

Model Context Protocol — What and Why

The Problem

AI coding tools are isolated. They can see your open files, maybe your repo, but they cannot talk to your database, your issue tracker, your design tool, or your deployment pipeline. Every integration requires custom glue code.

Here is what I would say: "Before MCP, getting AI to talk to your database or GitHub meant custom glue code for every integration. Every vendor had their own plugin system, their own auth story, their own data format. MCP changes that. Think of it as USB-C for AI tooling — one standard port that everything plugs into."

MCP as "Plugins for AI"

The Model Context Protocol is an open standard (by Anthropic) that provides a standardized way to connect AI assistants to external data sources and tools. Instead of building bespoke integrations, you configure a server and the AI discovers what it can do.

+------------------+ +------------------+ +------------------+ +---------------------+ | | | | | | | | | Cursor IDE | <---> | MCP Client | <---> | MCP Server | <---> | External Services | | (AI Agent) | | (built-in) | | (configured) | | GitHub, DB, Figma | | | | | | | | Filesystem, etc. | +------------------+ +------------------+ +------------------+ +---------------------+

Three Primitives

Tools

Functions the AI can call. Like "create a GitHub issue" or "run a SQL query". The AI decides when and how to invoke them.

Resources

Data the AI can read. Think database schemas, file contents, API documentation. Pulled in as context on demand.

Prompts

Reusable templates that combine instructions with context. Pre-built workflows the user or AI can trigger.

Configuring MCP Servers

The Configuration File

MCP servers are configured in .cursor/mcp.json at your project root. Cursor reads this on startup and connects to each server.

// .cursor/mcp.json { "mcpServers": { "filesystem": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/project"] }, "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_..." } }, "postgres": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://localhost/mydb"] } } }

Common MCP Servers

Server What It Does Package
Filesystem Read/write files outside the workspace @modelcontextprotocol/server-filesystem
GitHub Issues, PRs, repos, code search @modelcontextprotocol/server-github
Playwright Browser automation and testing @playwright/mcp
Postgres Query databases, inspect schemas @modelcontextprotocol/server-postgres
Memory Persistent key-value memory for the AI @modelcontextprotocol/server-memory

Pro Tip

Check MCP status any time with Cmd+Shift+P then type "MCP: Show Status". This shows which servers are connected, which tools are available, and any errors.

Figma MCP — Design to Code

Here is what I would say: "Figma MCP is where design-to-code stops being a hand-off and starts being a conversation. Instead of a designer exporting a spec and a developer squinting at pixels, the AI agent reads the design directly and generates matching components. The Figma file becomes a live source of truth."

What Figma MCP Enables

  • Extract design context — AI reads colors, typography, spacing, layout from Figma frames
  • Generate code from frames — Copy a Figma frame URL, paste it into the Cursor agent, get a matching React component
  • Code Connect — Map Figma components to your existing codebase components so the AI reuses your design system instead of generating from scratch

Two Server Options

Remote Server (Recommended)

Connects to Figma's hosted MCP endpoint. No local install required. Works with any OS.

// .cursor/mcp.json { "mcpServers": { "figma": { "url": "https://mcp.figma.com/mcp" } } }

Desktop Server

Runs locally via the Figma desktop app. Requires Figma desktop to be running. Better for offline use or enterprise environments.

// .cursor/mcp.json { "mcpServers": { "figma": { "command": "npx", "args": [ "-y", "figma-developer-mcp", "--stdio" ] } } }

The Workflow

  1. Open your Figma design

    Navigate to the frame or component you want to implement.

  2. Copy the frame URL

    Right-click the frame in Figma and select "Copy link". This gives you a URL like https://figma.com/design/FILE_KEY/...?node-id=123-456.

  3. Paste into Cursor Agent

    In Cursor's agent chat, paste the URL and describe what you want: "Generate a React component matching this Figma frame using our design system tokens."

  4. AI reads the design and generates code

    The agent calls the Figma MCP tools to extract layout, styles, and content, then produces a component that matches the design.

Best Practices

  • Clean Figma files — use Auto Layout, name your layers, keep structure tidy. The AI reads your layer names.
  • Design system hygiene — use shared styles and components in Figma so the AI can map them to your code tokens.
  • Code Connect — set up component mappings so the AI generates <Button variant="primary"> instead of raw HTML with inline styles.
  • Frame-level prompts — point at specific frames, not entire pages. The more focused the context, the better the output.

Plan Requirements

Figma MCP requires a Full or Dev seat. Call limits: Professional plan = 200 calls/day, Enterprise plan = 600 calls/day. Free and Starter plans do not have MCP access.

Custom Slash Commands

What They Are

Plain markdown files stored in .cursor/commands/. Each file becomes a slash command you can invoke in the agent chat. The filename (minus extension) becomes the command name.

File Structure

.cursor/ commands/ review.md # /review add-endpoint.md # /add-endpoint fix-types.md # /fix-types

Example: /review

# .cursor/commands/review.md Review the selected code for: 1. Security issues (SQL injection, XSS, auth) 2. Performance (N+1 queries, memory leaks) 3. Readability (naming, complexity, DRY) 4. Testing gaps Be specific. Cite line numbers. Suggest fixes.
Here is what I would say: "Commands are the simplest primitive. One file, one action. If you find yourself typing the same prompt over and over, turn it into a command. It takes 30 seconds and your whole team gets it via version control."

Skills with SKILL.md

How Skills Differ from Commands

Commands are single actions. Skills are multi-step workflows with supporting files. A skill can include shell scripts, templates, validation logic, and step-by-step instructions for the agent to follow.

# .cursor/skills/add-api-endpoint/SKILL.md --- name: add-api-endpoint description: Scaffold a new REST endpoint with controller, service, repository, and tests --- # Add API Endpoint ## Steps 1. Ask the user for the resource name (e.g., "Order") 2. Create the JPA entity in `backend/src/main/java/models/` 3. Create the repository interface 4. Create the service class with CRUD methods 5. Create the REST controller with standard endpoints 6. Create the unit test file 7. Run `./mvnw compile` to verify 8. Report what was created ## Templates Use the template files in this skill's directory for consistent formatting.

Supporting Files

Skills can include anything in their directory — shell scripts, code templates, configuration files. The agent uses these as context when executing the skill.

.cursor/skills/add-api-endpoint/ SKILL.md # The skill definition templates/ Entity.java.template # JPA entity boilerplate Controller.java.template Service.java.template scripts/ validate.sh # Post-creation validation

Create Skills Interactively

Use /create-skill in Cursor's agent chat. It walks you through naming, describing, and scaffolding a new skill directory.

Migrate Existing Commands

Already have complex commands that should be skills? Use /migrate-to-skills to convert them. It preserves your prompt content and adds the YAML frontmatter.

Custom Subagents

What Are Subagents?

Specialized AI agents that the main agent can delegate to. Each subagent has its own personality, permissions, and model configuration. Defined as markdown files in .cursor/agents/.

# .cursor/agents/security-reviewer.md --- name: security-reviewer description: Reviews code changes for security vulnerabilities, OWASP Top 10, and auth issues model: claude-sonnet readonly: true is_background: false --- You are a security-focused code reviewer. Analyze all code changes for: - SQL injection and parameterized query usage - XSS vulnerabilities in frontend code - Authentication and authorization gaps - Secrets exposure (API keys, passwords in code) - Input validation and sanitization - CSRF protection Never approve code with known security issues. Be specific about line numbers and remediation.

YAML Frontmatter Fields

Field Type Purpose
name string Identifier, used for invocation
description string What this agent does. The main agent reads this to decide when to delegate.
model string Which LLM model to use (e.g., claude-sonnet, gpt-4o)
readonly boolean If true, the subagent cannot modify files — analysis only
is_background boolean If true, runs in an isolated cloud environment with git worktree

Auto-Delegation & Built-in Subagents

The main agent reads each subagent's description and automatically delegates when a task matches. You can also explicitly invoke with /agent-name.

Explore

Built-in. Searches and reads code across the repo. Read-only. Great for understanding unfamiliar codebases.

Bash

Built-in. Executes shell commands. Used for builds, tests, package installs, and system operations.

Browser

Built-in. Navigates web pages, takes screenshots, interacts with UI. Powers visual testing and web research.

Parallel Execution & Isolation

Background subagents (is_background: true) run in parallel using git worktree isolation. Each gets its own branch and working directory so they cannot conflict with each other or your main workspace. Up to 8 background agents can run simultaneously.

SESSION 4 — 10:45 - 12:00

Playwright + MCP Testing

Playwright MCP for AI-Driven Testing

Here is what I would say: "Playwright MCP gives the AI agent a real browser. Not a simulated one, not a screenshot — an actual Chromium instance it can navigate, click, type into, and inspect. You describe what you want tested, and the agent writes and runs the test in one shot."

What Playwright MCP Adds

With the Playwright MCP server connected, the AI gains browser automation tools: navigate to URLs, click elements, fill forms, take screenshots, assert on page content. These are exposed as MCP tools the agent can call during conversation.

Configuration

// .cursor/mcp.json { "mcpServers": { "playwright": { "command": "npx", "args": ["-y", "@playwright/mcp"] } } }

The Workflow

  1. Describe the test scenario

    "Test that a user can add a product to the cart and see the correct total at checkout."

  2. Agent writes the Playwright test

    The AI generates a full .spec.ts file using your Page Object Model, data-testid selectors, and assertion patterns.

  3. Agent runs the test

    The agent executes npx playwright test and reports results. If it fails, the agent reads the error output and iterates.

Page Object Model Pattern

The Page Object Model (POM) keeps selectors and page interactions in reusable classes. This is the pattern we follow in ShopCursor's E2E tests:

// e2e/pages/ProductPage.ts export class ProductPage { constructor(private page: Page) {} async goto() { await this.page.goto('/products'); } async addToCart(productName: string) { const card = this.page.locator(`[data-testid="product-card"]`) .filter({ hasText: productName }); await card.locator('[data-testid="add-to-cart"]').click(); } async getCartCount() { return this.page.locator('[data-testid="cart-count"]').textContent(); } }

Generating and Debugging E2E Tests

AI-Generated Tests from Natural Language

Instead of writing E2E tests by hand, describe what you want tested. The AI produces idiomatic Playwright code that follows your project conventions.

You Say

"Write an E2E test that: 1. Goes to the product page 2. Adds the Git Commit Hoodie to cart 3. Navigates to the cart 4. Verifies the item appears 5. Updates quantity to 2 6. Verifies the total updates"

AI Produces

test('cart quantity update', async ({ page }) => { const productPage = new ProductPage(page); const cartPage = new CartPage(page); await productPage.goto(); await productPage.addToCart('Git Commit Hoodie'); await cartPage.goto(); await expect(cartPage.itemRow('Git Commit Hoodie')) .toBeVisible(); await cartPage.updateQuantity('Git Commit Hoodie', 2); await expect(cartPage.total()) .toContainText('$79.98'); });

Using data-testid Selectors

Always prefer data-testid attributes over CSS classes or text content. They are stable across refactors and make tests resilient to UI changes.

<!-- In your React component --> <button data-testid="add-to-cart">Add to Cart</button> // In your Playwright test await page.locator('[data-testid="add-to-cart"]').click();

Debugging Failing Tests with Cursor

  • Paste the error output into Cursor chat — the AI reads the stack trace and suggests fixes
  • Use trace viewer — npx playwright show-trace trace.zip for step-by-step screenshots
  • Ask the agent to add waits — most flaky tests need waitForSelector or waitForResponse
  • Run headed mode — npx playwright test --headed to watch the browser in real time

AI Governance & Hooks

Agent Hooks for Governance

Here is what I would say: "Hooks are your safety net. The AI agent is powerful, but unchecked power is a liability. Hooks let you inject guardrails at every stage — before a file is read, after a file is edited, before a shell command runs. Think of them as middleware for agent actions."

Configuration

Hooks are defined in .cursor/hooks.json. Each hook binds a script to a lifecycle event.

// .cursor/hooks.json { "version": 1, "hooks": { "afterFileEdit": [{ "command": ".cursor/hooks/format.sh", "timeout": 10, "failClosed": false }], "beforeShellExecution": [{ "command": ".cursor/hooks/block-dangerous.sh", "timeout": 5, "failClosed": true }], "beforeReadFile": [{ "command": ".cursor/hooks/redact-secrets.sh", "matcher": "**/.env*", "timeout": 5, "failClosed": true }] } }

Lifecycle Events

Event When It Fires Use Case
afterFileEdit After the agent modifies any file Auto-format, lint, validate
beforeShellExecution Before the agent runs a terminal command Block dangerous commands
beforeReadFile Before the agent reads a file into context Redact secrets, filter sensitive data
sessionStart When an agent session begins Audit logging, setup
sessionStop When an agent session ends Cleanup, final audit entry

Hook Responses

allow

Let the action proceed. The hook ran successfully and found no issues.

deny

Block the action entirely. The agent is told the action was denied and why.

ask

Pause and ask the user for approval. The hook provides context about what it found.

Configuration Options

  • timeout — Max seconds the hook script can run before being killed
  • failClosed — If true, a hook timeout or error blocks the action. If false, the action proceeds.
  • matcher — Glob pattern to limit which files trigger the hook

Priority Order

Hooks are evaluated in priority order: Enterprise → Team → Project → User. Enterprise hooks cannot be overridden by project-level hooks. This is how organizations enforce mandatory governance.

Practical Hook Examples

Auto-Format After Edits

Run Prettier or your formatter automatically after every file the agent touches.

#!/bin/bash # .cursor/hooks/format.sh # Receives the edited file path as $1 FILE="$1" case "$FILE" in *.ts|*.tsx|*.js|*.jsx|*.json|*.css) npx prettier --write "$FILE" echo '{"action": "allow", "message": "Formatted with Prettier"}' ;; *.java) # Use google-java-format or spotless ./mvnw spotless:apply -q echo '{"action": "allow", "message": "Formatted Java code"}' ;; *) echo '{"action": "allow"}' ;; esac

Block Dangerous Commands

Prevent the agent from running destructive shell commands.

#!/bin/bash # .cursor/hooks/block-dangerous.sh # Receives the command as $1 CMD="$1" # Block destructive patterns if echo "$CMD" | grep -qE 'rm -rf /|drop database|git push.*--force|:(){ :|:& };:'; then echo '{"action": "deny", "message": "Blocked: destructive command detected"}' exit 0 fi # Flag production-touching commands for approval if echo "$CMD" | grep -qE 'prod|production|deploy'; then echo '{"action": "ask", "message": "This command references production. Proceed?"}' exit 0 fi echo '{"action": "allow"}'

Secrets Redaction

#!/bin/bash # .cursor/hooks/redact-secrets.sh # Redact .env values before LLM sees them FILE="$1" REDACTED=$(sed -E 's/(=).+/=\[REDACTED\]/' "$FILE") echo "{\"action\": \"allow\", \"content\": \"$REDACTED\"}"

Audit Logging

#!/bin/bash # .cursor/hooks/audit-log.sh # Log all shell commands the agent runs CMD="$1" TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ") USER=$(whoami) echo "$TIMESTAMP | $USER | $CMD" >> .cursor/audit.log echo '{"action": "allow"}'

Project Rules for Quality

What Are Rules?

Rules are persistent instructions that shape how the AI writes code in your project. Defined in .cursor/rules/*.mdc with YAML frontmatter that controls when they activate.

# .cursor/rules/java-standards.mdc --- description: Java coding standards for the ShopCursor backend globs: ["backend/**/*.java"] --- ## Java Standards - Use constructor injection, never field injection with @Autowired - Return Optional<T> from repository methods, never null - Use @Valid on request body parameters - Log at appropriate levels: ERROR for failures, INFO for business events, DEBUG for diagnostics - Write unit tests for all service methods - Use descriptive variable names, not single letters

Four Activation Modes

Mode Frontmatter Behavior
Always Apply alwaysApply: true Included in every prompt, every time. Use sparingly.
Intelligent description: "..." only The agent reads the description and decides if it is relevant to the current task.
Glob-based globs: ["**/*.java"] Automatically included when working with files that match the pattern.
Manual No auto-trigger fields Only activated when the user explicitly @-mentions the rule in chat.

Create Rules Interactively

Use /create-rule in the agent chat. It walks you through choosing a name, mode, and content for the rule, then writes the .mdc file for you.

Layered Governance: Rules + Hooks + Subagent Readonly

These three primitives combine into a layered governance model:

  1. Rules

    Prevention: Tell the AI how to write code correctly in the first place. Standards are embedded in every prompt.

  2. Hooks

    Enforcement: Validate after the fact. Auto-format edited files, block dangerous commands, redact secrets. The safety net.

  3. Subagent Readonly

    Isolation: Some agents should only analyze, never modify. Set readonly: true to ensure a reviewer subagent cannot accidentally change code.

Security Settings & Data Leakage Prevention

Here is what I would say: "Everything we have covered so far is about securing the code the AI writes. But there is a second, equally important question: what data are you sending to the AI? Every prompt, every file the agent reads, every terminal output — all of it leaves your machine and travels to an LLM provider. If you are working on proprietary code, client projects, or anything with PII, you need to understand and control that data flow."

What Data Leaves Your Machine?

When you use Cursor, the following data may be sent to external servers:

Data Type When It Is Sent Where It Goes
Code in open files Every prompt, autocomplete, Cmd+K LLM provider (OpenAI, Anthropic, or Cursor API)
Codebase index When indexing is enabled Cursor’s servers (embeddings)
Terminal output Agent mode shell commands LLM provider
File contents @file references, agent reads LLM provider
Error messages & stack traces Debugging prompts LLM provider
Chat history Ongoing conversation context LLM provider

The Core Risk

Proprietary source code, API keys, database credentials, PII, and internal business logic can all leak to third-party servers through normal Cursor usage. This is not a bug — it is how AI coding assistants work. Your job is to control what gets sent.

Privacy Mode

Cursor’s most important security setting. Found in Settings → Privacy.

Setting Behavior
Privacy Mode: OFF (default) Code snippets may be stored by LLM providers for service improvement
Privacy Mode: ON Code is sent for inference only — not stored, not used for training

Best Practice

Always enable Privacy Mode for proprietary or client code. With Privacy Mode on, Cursor states that no code is retained by any third party. Enable via Settings → Privacy → toggle ON, or set "cursor.privacy.mode": true in settings.

Codebase Indexing

When enabled, Cursor sends file contents to its servers to build a semantic search index for @codebase queries.

Setting What Happens
Indexing ON Files are chunked, embedded, and stored on Cursor’s servers
Indexing OFF No codebase data leaves your machine (but @codebase will not work)

For repositories containing sensitive IP or customer data, disable indexing entirely in Settings → Features → Codebase Indexing.

Telemetry

Controls whether usage analytics and crash reports are sent to Cursor.

  • Disable in Settings → Privacy → toggle telemetry OFF
  • Or set "telemetry.telemetryLevel": "off" in settings
  • Required for SOC 2, HIPAA, and other compliance frameworks

.cursorignore — Your Most Important Defense

Here is what I would say: "If Privacy Mode is the switch, .cursorignore is the scalpel. It tells Cursor: never read this file, never index it, never send it to any LLM, period. This is your single most effective control against data leakage."

.cursorignore vs .cursorindexingignore

File LLM Sees It? Indexed for @codebase? Use For
.cursorignore Never Never Secrets, credentials, PII, proprietary algorithms
.cursorindexingignore Yes (if opened) Never Large build artifacts, binaries, generated files

Recommended .cursorignore

# Secrets and credentials .env .env.* *.pem *.key *.p12 *.jks credentials.json service-account.json **/secrets/ # Infrastructure terraform.tfstate terraform.tfstate.backup *.tfvars # Production configs **/application-prod.* **/application-staging.* # Client / proprietary data **/data/customers/ **/data/exports/ **/fixtures/real-data/ # MCP config (may contain API tokens) .cursor/mcp.json # Internal documentation docs/internal/ docs/architecture/threat-model*

Commit It to Source Control

Always commit .cursorignore to your repository so every team member gets the same protection. Treat it like .gitignore — it is a shared security boundary.

MCP & Agent Mode Security

Here is what I would say: "MCP servers give the agent superpowers — it can read your filesystem, call GitHub APIs, query databases. But every tool you grant is an expansion of the attack surface. And in Agent mode, the AI runs commands autonomously. You need to think about what you are allowing it to access."

MCP Server Risks

Token Leakage

API tokens hardcoded in .cursor/mcp.json can appear in error messages sent to the LLM.

Over-Permissioned Tools

A filesystem MCP with access to ~/ can read ~/.ssh/, ~/.aws/, and other sensitive directories.

Unreviewed Actions

The agent can call MCP tools that modify external systems — creating PRs, sending messages, altering databases.

MCP Best Practices

  1. Scope Filesystem Access Narrowly

    Only grant access to the directories the agent needs. Never use ~/ or /.

    // Good: scoped to project source directories "args": ["-y", "@modelcontextprotocol/server-filesystem", "./backend/src", "./frontend/src"] // Bad: access to entire home directory "args": ["-y", "@modelcontextprotocol/server-filesystem", "~/"]

  2. Use Environment Variables for Tokens

    Never hardcode API keys in mcp.json. Reference environment variables instead.

    // Good: token from environment "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } // Bad: token in plain text in a committed file "env": { "GITHUB_TOKEN": "ghp_abc123..." }

  3. Audit MCP Tool Calls

    Use a hook on beforeMCPExecution to log every MCP tool invocation for compliance.

  4. Review Third-Party MCP Servers

    Read the source code of any community MCP server before adding it. Verify what data it accesses and where it sends it.

Agent Mode Permission Settings

In Settings → Features → Agent, control how much autonomy the agent has:

Setting Behavior Risk Level
Always allow Agent runs commands without asking High
Ask for approval Agent pauses for your confirmation before shell commands Recommended
Never allow Agent cannot run shell commands at all Safest, but limits capability

Review Every Command

Even with "Ask for approval", do not blindly click "Allow". Read the command. Watch for curl with tokens in URLs, git push to unexpected remotes, npm install of unknown packages, and database commands against production.

Common Data Leakage Scenarios

How Leaks Happen

Scenario How It Happens Prevention
Secrets in autocomplete Cursor reads .env to provide context .cursorignore
API key in error message Agent runs a failing curl with token in the URL Secrets redaction hook
Proprietary algorithm indexed Codebase indexing sends embeddings to Cursor .cursorindexingignore or disable indexing
PII in test fixtures Agent reads fixtures/users.json with real customer data .cursorignore, use synthetic data
Credentials in MCP config Hardcoded tokens in mcp.json visible to the agent Use env vars, add to .cursorignore
Internal docs in prompt Developer pastes confidential document into chat Training & awareness
Production DB credentials Agent reads application-prod.yml .cursorignore, separate prod config

Team & Enterprise Security Checklist

Before Onboarding

  • Enable Privacy Mode for all team members
  • Create .cursorignore in every repository
  • Create .cursorindexingignore for build artifacts
  • Audit .cursor/mcp.json for hardcoded secrets
  • Set up block-dangerous.sh hook
  • Set up secrets redaction hook

Ongoing Governance

  • Review agent activity logs periodically
  • Rotate API keys that may have been exposed
  • Disable codebase indexing for sensitive repos
  • Pin MCP server versions (no latest)
  • Educate: never paste secrets or PII into chat
  • Disable telemetry if required by compliance

Network & Compliance

  • Allowlist endpoints: api2.cursor.sh, api.openai.com, api.anthropic.com
  • Proxy LLM traffic through corporate proxy for logging if required
  • Document data flow: where code goes, which providers, retention policies
  • Legal review: Review Cursor’s privacy policy and ToS with your compliance team

The Complete Defense Stack

Privacy Mode (no data retention) + .cursorignore (block sensitive files) + Hooks (redact secrets, block commands) + Rules (guide AI to write secure code) + Training (developer awareness). Technical controls alone are not enough — your team needs to understand the risks.

Code Review for AI-Generated Code

Here is what I would say: "Here is the mindset shift: treat AI-generated code like a junior developer's pull request. It is usually correct, often clever, sometimes subtly wrong, and occasionally dangerous. Your job is not to rubber-stamp it — your job is to review it with the same rigor you would apply to any human-written code."

Security Checklist

SQL Injection

Is the AI using parameterized queries? Or is it concatenating strings into SQL? Check every database call.

Authentication

Are new endpoints protected? Did the AI add proper auth checks? Does it handle token expiry?

Input Validation

Is user input validated and sanitized? Are there length limits, type checks, and allowlist patterns?

Anti-Patterns to Watch For

  • God classes — AI loves putting everything in one file. Watch for 500+ line classes.
  • Over-engineering — Abstract factory for a feature with one implementation. Three layers of indirection for a simple CRUD.
  • Copy-paste — AI sometimes duplicates logic instead of extracting shared methods. Check for near-identical code blocks.
  • Hardcoded values — Magic strings, URLs, timeouts baked into code instead of configuration.
  • Missing error handling — Happy path works great. What happens when the database is down?

BugBot Automated PR Review

BugBot is Cursor's automated PR reviewer. It runs on every pull request and comments on potential issues.

  • Scans for security vulnerabilities, anti-patterns, and style violations
  • Provides inline comments on specific lines
  • Includes a "Fix in Cursor" button that opens the issue directly in your IDE with the fix pre-loaded
  • Configurable sensitivity and rule sets per repository

The Reviewer Mindset

Treat AI code like a junior developer's PR. It is usually correct, often clever, sometimes subtly wrong, and occasionally dangerous. Never rubber-stamp. Always review with the same rigor you apply to human-written code. The AI does not know your business domain — you do.
